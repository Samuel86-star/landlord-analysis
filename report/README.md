## 跨层交叉要点

三份报告独立呈现各自层级的发现，以下是跨层才能看清的关键洞察：

### 1. "玩法维度" 和 "客户端维度" 要一起看

- by-mode 发现**不洗牌是留存最高的玩法**（同玩法次留 20.56% > 经典 17.81%）
- by-client-lang 发现 **Creator 集中在 iOS、Lua iOS 塌陷**
- **未交叉**：不洗牌玩法在 Creator 和 Lua 上的表现是否一致？如果 Lua iOS 问题也出现在不洗牌上，可能是该玩法依赖某个 Lua 特性
- 建议：补做 `玩法 × 客户端` 二维交叉，验证两类差异是否独立

### 2. 留存最强的杠杆是首日对局量

- global：10 局以上次留 34.9%、30 分钟+ 次留 44.9%（vs 1 局用户 9.57%）
- by-mode：多玩法探索者（3 种玩法）次留 33.23%（vs 单玩法 21.73%）
- **共同指向**：**让新用户在首日"玩够"是最大杠杆**——玩够局数、玩够时间、玩够玩法
- 但要警惕因果倒置：可能是"会留存的人本来就愿意多玩"。建议 A/B 测试：连败保护 / 任务引导 / 新手礼包按对局数阶梯发放

### 3. 连败是跨层一致的流失预警，但阈值不同

- global：连败 2 局次留仅 4.80%，是毁灭性打击
- by-mode：所有玩法都显示连败越多次留越高（6-9 连败 33-38%）——这个**与全局结论矛盾**

矛盾的原因很可能是：by-mode 的分母是"某玩法内有对局的用户"，而 global 分母是"所有新增用户"。by-mode 连败 6-9 局的用户本身就是"玩了很多局"的活跃用户，所以留存自然高。
**结论**：以 global 的连败分析为准（连败 2 局是预警线），by-mode 的连败分析需重新设计（控制对局数后再看连败的独立效应）。

### 4. 数据质量问题跨层共现

| 问题 | 出现位置 | 应对 |
|------|---------|------|
| 咪咕渠道异常低留存（5.34%） | global 8.2.2 | 建议降级或切断 |
| 癞子 360/努比亚渠道异常（1-2%） | by-mode | 疑似机器流量，需核查 |
| L-09 版本切换全量"未切换" | by-client-lang | SQL/表结构需重审 |

**非数据质量问题的业务事实**：
- 新增用户首日 100% 无逃跑（global 8.3.7 / by-client-lang L-12）——已验证是真实事实：新手礼包+新手配牌+默认经典玩法的保护下，新增用户没有弃局动机；挫败反馈走"不登录"而非"局内逃跑"。**逃跑率对新增用户阶段的分析无诊断价值**，不必排查 escape_count 聚合逻辑

## 建议行动顺序

### P0（数据最强证据支持，可立即启动）

1. **新手连败保护**：连败 2 局即触发匹配降难度 / 安慰银子（依据：global 连败分析）
2. **咪咕渠道降级**：留存是基线的 1/4，不降级则继续拉低整体指标
3. **Lua iOS 排查**：iOS 侧两版本差距 2.3 倍，而 Creator iOS 正常——非业务问题，技术侧排查

### P1（需 A/B 测试验证）

4. **首日任务引导对局数**：玩够 5/10 局送奖励（依据：对局数强正相关 + 癞子"1 局即走"问题）
5. **玩法探索引导**：经典用户推荐尝试不洗牌（依据：不洗牌是潜力玩法）
6. **差异化倍数策略**：不洗牌/癞子不设倍数上限（依据：它们的留存随倍数单调递增）

### P2（数据缺口先补齐）

7. 补 L-09 版本切换分析、重构 by-mode 连败分析（控制对局数）、从监控看板移除新手逃跑率指标
8. **首日仅 1 局专项拆解**：按 [`../docs/one-game-first-day-analysis-plan.md`](../docs/one-game-first-day-analysis-plan.md) 执行渠道/首局/时长等归因，支撑 P1「首日任务引导对局数」的优先级与实验设计
